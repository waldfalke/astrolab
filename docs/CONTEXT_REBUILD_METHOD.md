# Пересобранный контекст методологии

## 1) Что уже согласовано

1. Рабочее пространство модели — **одномерный цикл** (`S1`, 360°), а не физическое 3D-описание.
2. Сведение к окружности и 360° — это **нормализация** для вычислений состояний.
3. Базовая логика — **фазовые сдвиги** (а не нарративные интерпретации).
4. Нужны:
   - матрица состояний,
   - граф диспозиций,
   - единая нотация для направленных связей,
   - поддержка знаков + домов (Плацид),
   - микро-фазы.
5. Стиль результата: нейтрально, без давления/внушения.

## 2) Базовые сущности

1. **Функция**: планета как оператор процесса.
2. **Фаза**: позиция от 1 до 12.
3. **Материнская точка**: обитель функции (база для фазового отсчета по знаку).
4. **Сцена реализации**: дом (для событийного слоя).
5. **Диспозитор**: управитель знака положения планеты.

## 3) Матрица `12x12`

- Строки: функции (обычно 10 планет).
- Столбцы: фазы `1..12`.
- Ячейка = состояние функции в данной фазе.

Фазовый словарь (единый для знаков и домов):

1. Инициация / Self  
2. Ресурс / Питание  
3. Связь / Передача  
4. База / Основание  
5. Выход / Творчество  
6. Сервис / Настройка  
7. Интерфейс / Другой  
8. Кризис / Пересборка  
9. Стратегия / Смысл  
10. Результат / Цель  
11. Оптимизация / Будущее  
12. Завершение / Фон

## 4) Векторы состояния планеты

Для каждой планеты фиксируется минимум три слоя:

1. `Z` — фаза по знаку относительно собственной обители.
2. `H` — фаза по дому относительно ASC (в выбранной системе домов).
3. `D` — фаза диспозитора (состояние управляющей планеты).

Рабочий код состояния:

`P = (Z, H, D)`

## 5) Микро-фазы

### 5.1 По знаку

- Каждый знак (30°) делится на 12 равных подэтапов по `2.5°`.
- Микро-фаза: `z = ceil(deg_in_sign / 2.5)`, диапазон `1..12`.
- Используется `ceil`, чтобы избежать «нулевой фазы» и фиксировать активное состояние после входа в сегмент.

### 5.2 По дому (Плацид)

- Дом неравномерен, поэтому деление не по фиксированным градусам, а по доле прохождения внутри дома.
- `L_house = cusp_next - cusp_current` (по кругу).
- `L_pos = lon_planet - cusp_current` (по кругу).
- `h = ceil(12 * L_pos / L_house)`, диапазон `1..12`.

Итоговый расширенный код:

`P = (Z.z, H.h, D)`

## 6) Граф диспозиций (иерархия управления)

1. Узлы: планеты.
2. Направленное ребро: `A -> B`, где `B` — диспозитор `A`.
3. Подпись ребра — фазовый индекс `A` относительно своей обители:
   - пример: `Mars -(7)-> Venus`.
4. Центр/аттрактор: узел(ы), куда стекаются цепочки управления (часто планета в обители или замкнутый контур).

## 7) Аспекты в нотации

Принята идея направленной записи (для вычислительной модели), с разделением:

- верхний индекс: фазовый смысл связи,
- нижний индекс: геометрический угол.

Схема:

`A ---(верхний_индекс)---> B`
`           нижний_угол`

Важно из лога:

1. Пользователь отдельно зафиксировал: не использовать симметричное `<->` как общую норму.
2. Исключения и строгие правила коммутативности для аспектов должны быть явно прописаны в финальной спецификации отдельно.

## 8) Как читать «знак + дом» вместе

1. `Z` отвечает на «**как** функция работает по природе».
2. `H` отвечает на «**где** функция реализуется событийно».
3. `D` отвечает на «**кто санкционирует/перенаправляет** функцию».
4. Микро-уровень (`z`, `h`) уточняет текущую стадию внутри этапа.

## 9) Что осталось незакрытым

1. Финально утвердить строгое правило направленности аспектов для всех типов.
2. Зафиксировать канон управителей (традиционные/современные) без двусмысленностей.
3. Развести «иллюстративные» и «верифицированные» расчеты (пример из лога содержит спорные числа).
4. Собрать формальную спецификацию в виде:
   - словаря терминов,
   - формул,
   - формата записи,
   - тест-кейсов.

## 10) Практический следующий артефакт

Собрать `SPEC.md` с 4 разделами:

1. Формальная математика (`Z, H, D, z, h`).
2. Нотация графа (`disposition`, `aspect`).
3. Протокол вычисления по шагам.
4. Набор контрольных примеров с ожидаемым выводом.
