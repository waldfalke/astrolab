  1. polar-mcp (jamaljsr)

  Что это: MCP-сервер для управления тестовыми Lightning-сетями через Polar.

  Инструменты: 40+ tools в 7 категориях — управление сетями, Bitcoin-операции
  (майнинг блоков), Lightning-операции (каналы, инвойсы, платежи), Taproot
  Assets.

  Поддерживает: LND, c-lightning, Eclair, litd, tapd — всё что есть в Polar.

  КРИТИЧЕСКИЙ БЛОКЕР: Требует запущенный Polar GUI (десктоп-приложение).
  Общается с ним через HTTP bridge на localhost:37373. Headless/Docker
  невозможно. В CI/CD — нет.

  Вердикт для нас: На Windows с десктопом — можно использовать для локальной
  разработки и тестирования. Но это зависимость от GUI-приложения, что
  противоречит нашей headless-философии. Для CI/CD не годится. Полезен как 
  dev-tool, но не как production-компонент.

  ---
  2. refined-element/lightning-enable-mcp

  Что это: Полноценный L402-клиент и сервер. 15 tools. Две реализации: .NET и
  Python.

  Ключевые возможности:
  - Оплата BOLT11 инвойсов
  - Создание инвойсов
  - Автоматический L402-flow — обнаружение L402-manifest, автооплата
  402-челленджей
  - Бюджетирование и трекинг расходов
  - On-chain отправка

  Подключение к LND: REST + macaroon. Также поддерживает Strike, NWC (CoinOS,
  Alby Hub), OpenNode.

  Docker: Есть.

  Вердикт для нас: ЭТО ЗОЛОТО. Этот MCP делает ровно то, что нам нужно:
  - Серверная сторона — создание инвойсов, верификация оплаты → заменяет нашу
  ручную gRPC-логику в mock-api
  - Клиентская сторона — автооплата L402 → заменяет нашу ручную логику в
  client-agent
  - L402-прокси — можно обернуть любой API и добавить платёж → ровно наш Этап 2

  ---
  3. getAlby/mcp

  Что это: MCP-сервер для подключения Lightning-кошелька к LLM через NWC (Nostr
  Wallet Connect).

  Подключение: Только через NWC-протокол. Нужен NWC-совместимый кошелёк (Alby
  Hub, и т.д.).

  Проблема: Не работает напрямую с LND gRPC. Привязан к экосистеме Alby/NWC.
  Документация слабая — tools не перечислены, regtest не упоминается.

  ehallmark/btc-lightning-client: Python-библиотека для прямого gRPC к LND. Есть
   invoice listener. Но это библиотека, не MCP-сервер (MCP-обёртка в отдельном
  репо).

  Вердикт: getAlby слишком завязан на NWC/Alby экосистему. ehallmark — интересен
   как запасной вариант для low-level контроля, но не MCP-сервер сам по себе.

  ---
  Серверный L402 gate — что нашлось

  1. Aperture (Lightning Labs) — ГЛАВНЫЙ КАНДИДАТ

  Что это: Production-grade L402 reverse proxy от Lightning Labs. Go-бинарник,
  MIT.

  Как работает:
  - Стоит перед твоим API как reverse proxy
  - Запрос → Aperture → 402 + macaroon + Lightning invoice → клиент платит →
  Aperture верифицирует preimage → проксирует запрос к backend

  Конфиг для нашего случая будет выглядеть примерно так:
  authenticator:
    network: "regtest"           # поддерживает regtest/simnet!
    lndhost: "lnd:10009"
    tlspath: "/root/.lnd/tls.cert"
    macdir: "/root/.lnd/data/chain/bitcoin/regtest"

  services:
    - name: "astro-api"
      hostregexp: '.*'
      pathregexp: '^/sun_sign.*$'
      address: "mock-api:3000"
      protocol: http
      price: 1000                # 1000 sats за вызов

  Плюсы:
  - Полный L402-flow из коробки (macaroon + invoice + verify)
  - Поддержка regtest/simnet
  - Rate limiting, dynamic pricing, capabilities в macaroons
  - Есть Dockerfile
  - Production-ready (используется в Lightning Loop)

  Минусы:
  - Go-бинарник (нужно билдить или использовать Docker image)
  - Нет официального Docker image на Docker Hub (но есть Dockerfile в репо)

  2. PayGated — НЕ подходит

  Кредитная система + Stripe. Не L402, не Lightning.

  3. SatGate MCP Proxy — НЕ подходит

  Бюджетный прокси для клиента, не серверный gate. Не создаёт инвойсы.

  4. lightningfaucet/lightning-wallet-mcp — Интересен, но SaaS

  40+ tools, умеет L402. Но это hosted-сервис (lightningfaucet.com), не
  self-hosted. Нет regtest.
