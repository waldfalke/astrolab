# [IN_PROGRESS][TASK][P1] L402-MVP-007 - Adapt client-agent for Real LND

## Goal
Адаптировать client-agent для реальной оплаты Lightning invoice через собственный LND-узел, замкнув полный L402-цикл: 402 -> pay -> preimage -> resource.

## Problem Statement
1. Текущий client-agent использует dummy preimage. Это не доказывает реальную оплату.
2. Один LND-узел не может оплатить свой собственный invoice. Для реального платежа нужны два узла с каналом.

## Scope
1. **Добавление `lnd-client` в `docker-compose.yml`:**
   - Второй LND-узел для клиентской стороны.
   - Healthcheck аналогичный серверному LND.
   - Монтирование отдельного volume `lnd_client_data`.
2. **Init-скрипт `setup-channel.sh`:**
   - Создание bitcoind wallet.
   - Майнинг 101+ блоков для maturity.
   - Отправка средств на lnd-client.
   - Открытие канала lnd-client -> lnd (server).
   - Подтверждение канала (майнинг блоков).
3. **Обновление `client-agent`:**
   - Подключение к `lnd-client` через gRPC (TLS + macaroon).
   - Парсинг invoice из заголовка `WWW-Authenticate`.
   - Оплата invoice через `sendPaymentSync`.
   - Извлечение preimage из ответа.
   - Формирование L402-заголовка с реальным preimage.
4. **Обновление `mock-api/server.js`:**
   - При наличии заголовка Authorization: парсинг token (r_hash) и preimage.
   - Проверка через `lookupInvoice` что invoice оплачен.
   - Возврат платного ресурса.

## Out of Scope
1. Автоматический запуск init-скрипта в docker-compose (ручной этап).
2. Multi-hop routing, channel management.
3. Mainnet.

## Done Definition
1. `docker compose up -d` поднимает bitcoind, lnd, lnd-client, mock-api.
2. После init-скрипта между lnd и lnd-client есть открытый канал с балансом.
3. `docker compose run --build client-agent` выполняет полный цикл: 402 -> оплата -> preimage -> ресурс.
4. В логах видно реальный Lightning payment, а не dummy.
