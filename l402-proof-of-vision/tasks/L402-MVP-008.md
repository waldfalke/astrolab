# [DONE][TASK][P1] L402-MVP-008 - Implement LND with Healthchecks

## Goal
Развернуть стек `bitcoind` + `lnd`, используя встроенные в Docker Compose механизмы `healthcheck` для решения проблемы "гонки состояний" и обеспечения стабильного запуска.

## Problem Statement
1.  Предыдущие попытки запуска `lnd` провалились, так как он стартовал раньше, чем `bitcoind` был готов принимать RPC-соединения.
2.  Использование `entrypoint.sh` для ожидания создало непредвиденные проблемы с правами доступа в кросс-платформенной среде (Windows/Linux).

## Scope
1.  **Обновление `docker-compose.yml`:**
    - Удалить все `entrypoint` и кастомные `Dockerfile` для `lnd`. Вернуться к использованию "чистого" образа `lightninglabs/lnd`.
    - Добавить секцию `healthcheck` для сервиса `bitcoind`. Healthcheck будет проверять доступность RPC-порта.
    - В сервисе `lnd` использовать опцию `depends_on` с условием `service_healthy`, чтобы он не стартовал, пока `healthcheck` `bitcoind` не пройдет.
    - Добавить `healthcheck` для самого `lnd`, который будет проверять, создан ли кошелек (`lncli getinfo`).
    - В сервисе `mock-api` использовать `depends_on` с `service_healthy` для `lnd`.
2.  **Обновление `mock-api/server.js`:**
    - Вернуть безопасную логику подключения с использованием `tls.cert` и `admin.macaroon`.
    - Убрать кастомную логику ожидания файлов, так как `healthcheck` в `docker-compose` теперь гарантирует их наличие.

## Out of Scope
1.  Создание `client-agent` (он будет использован позже).

## Done Definition
1.  `docker-compose.yml` обновлен с использованием `healthcheck`.
2.  Команда `docker compose up -d` успешно запускает `bitcoind`, дожидается его "здоровья", а затем успешно запускает `lnd`.
3.  `lnd` остается в рабочем состоянии и ожидает создания кошелька.
4.  Последующий запуск `mock-api` также проходит успешно, и в логах `mock-api` видно успешное соединение с `lnd` (после создания кошелька).
